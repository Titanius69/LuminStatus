# LuminStatus

A public status page for your Minecraft network, generated by the proxy itself.

When a network goes down, three things happen at once: the support channel
fills with "is it down?", staff stop working because they are answering that
question, and players assume nobody is looking. The outage is always shorter
than the trust it costs.

External uptime monitors only know whether the proxy port answers. They cannot
see that the survival backend is dead while the proxy cheerfully accepts
players and drops them in the lobby. The proxy is the only component that knows
the whole internal picture, which is why this is a Velocity plugin.

---

## Five minutes to a working page

1. Drop `LuminStatus-1.0.0.jar` into `plugins/` and start the proxy.
2. Look in `plugins/luminstatus/public/`. There is an `index.html` there
   already, with every server from your `velocity.toml` on it.
3. Serve that directory with any web server, or sync it somewhere public.

That is the whole setup. Nothing in `config.conf` has to be edited to get a
working page.

```bash
# One way to publish it: a cron job that pushes the export to GitHub Pages.
*/5 * * * * cd /srv/status && cp -r /opt/velocity/plugins/luminstatus/public/* . \
  && git add -A && git commit -qm "status" && git push -q
```

## What it measures

**Actively**, every 30 seconds: a server list ping to each backend, which
returns latency, player count and version in one round trip.

**Passively**, for free: connection failures the proxy sees while routing real
players. No extra packets, and it reacts faster than the poll interval.

A component has to fail three checks in a row before the page calls it down.
One dropped packet is not an outage.

### Four states, not two

| State | Meaning |
| --- | --- |
| Operational | Reachable, latency in its normal band |
| Degraded | Reachable but slower than its own median by 3x |
| Under maintenance | Announced downtime; never alerts |
| Partial / major outage | Some or all of the group is unreachable |

`Degraded` is the one that earns its keep. Players complain about lag far more
often than they complain about downtime, and a binary up/down page has nothing
to say about lag. Degraded is measured against each server's own median latency
rather than a fixed threshold, because 90 ms is normal for a box three
datacentres away and alarming for one in the same rack.

### What it does not measure

The proxy does not tick, so it has no TPS. Any status page showing you a proxy
TPS figure is showing you a number it made up. LuminStatus reports availability
and latency, which is what the proxy can actually see.

## Incidents from in-game

During an outage the staff who know what is happening are already logged in.
Asking them to open a web dashboard first is how status pages end up
unmaintained, so the whole incident workflow is a command:

```
/luminstatus incident create major_outage survival-1,survival-2 Database maintenance
/luminstatus incident update 20260721-001 identified Connection pool was exhausted
/luminstatus incident resolve 20260721-001 Pool resized, everything is back
```

Resolving an incident also lifts any maintenance flags it set.

## Publishing

Two modes, and the second one is the one you want.

**Built-in HTTP server** (`http.enabled = true`). One port, serves the page,
`status.json`, `feed.atom` and `/healthz`. Simple, but many hosts do not allow a
second open port.

**Static export** (on by default). Writes `index.html`, `status.json` and
`feed.atom` to a directory every minute. Point a web server at it, sync it to
object storage, or commit it to GitHub Pages. This mode is the default
recommendation for one reason: a status page hosted on the machine that just
went down is not a status page. Most projects skip this and it is the single
most important thing about a status page.

The exported HTML embeds the current data directly, so the page shows real
numbers even opened straight off disk with no server behind it.

## Privacy

The page is public, so it never emits addresses, internal host names, ports or
stack traces. Only display names, states, latencies and player counts leave the
proxy. Set `display-name` per server if the internal names themselves are
sensitive, and `hidden = true` for anything that should not appear at all.

## Commands and permissions

| Command | What it does |
| --- | --- |
| `/luminstatus` | Current state of every component, in chat |
| `/luminstatus reload` | Rereads `config.conf`; keeps the old one if it fails to parse |
| `/luminstatus refresh` | Rebuilds and re-exports the page immediately |
| `/luminstatus maintenance <server> <on\|off>` | Marks a server as planned downtime |
| `/luminstatus incident create <impact> <servers> <title>` | Opens an incident |
| `/luminstatus incident update <id> <stage> <message>` | Posts an update |
| `/luminstatus incident resolve <id> <message>` | Closes it |
| `/luminstatus incident list` | Lists everything recorded |

Aliases: `/lstatus`, `/netstatus`. Everything requires `luminstatus.admin`.

## Building

```bash
gradle build
# build/libs/LuminStatus-1.0.0.jar
```

Requires Gradle 8.5+ and JDK 17 or newer to build; Velocity 3.3.0 or newer to
run. No Gradle wrapper is committed — run `gradle wrapper` once if you want one.

There are no shaded dependencies. Velocity already ships Configurate, Adventure
and SLF4J, the JSON handling is about 200 lines in `util/Json`, and the web
server is the JDK's own. The result is a jar measured in kilobytes and zero
class loading conflicts with anything else on your proxy.

## Data on disk

Everything lives in `plugins/luminstatus/`:

- `config.conf` — configuration
- `history.json` — daily uptime counters, pruned to the retention window
- `incidents.json` — incident log
- `public/` — the static export

History stores counters per day, not individual samples, so a 50-server network
keeps 90 days of history in a few kilobytes. Both files are written
atomically: a crash mid-save leaves the previous file intact.
